SECTION 1: INTRODUCTION
================================================================================

1.1 PURPOSE
    This Software Requirements Specification defines the complete functional
    and non-functional requirements for PrepCreatine, an AI-powered exam
    preparation platform for Indian competitive exam students and any student
    preparing for any type of examination. This document is intended for:
      - Frontend developers (using FDD for implementation)
      - Backend developers (using BSDD for implementation)
      - QA engineers (using this document for test planning)
      - Product stakeholders (for scope agreement and sign-off)
      - Any AI system generating code for this project

1.2 SCOPE
    PrepCreatine is a web application and Android application that provides:
      a) AI-powered tutoring and concept explanation
      b) Personalized study roadmap generation
      c) Exam-style mock testing with real-time results
      d) Knowledge ingestion from student-provided notes, URLs, and PDFs
      e) Syllabus tracking and progress monitoring
      f) Community Q&A with mentor oversight
      g) Study gamification to address motivational challenges
      h) Analytics dashboards for students and mentors

    The system specializes in Indian competitive examinations including JEE,
    NEET, CUET, GATE, CAT, UPSC, and State PETs but is designed to support
    any examination type including school tests, semester exams, and any
    subject-based study session.

    OUT OF SCOPE for Version 1.0:
      - Live video tutoring or human tutor matching
      - Payment processing or subscription management
      - Native iOS application
      - Development/tech course content (web dev, data science) — waitlist only
      - Real-time collaborative study sessions
      - Regional language interfaces (English only for v1.0)

SECTION 2: OVERALL DESCRIPTION
================================================================================

2.1 PRODUCT PERSPECTIVE
    PrepCreatine is a standalone web application with an Android companion.
    It integrates with the following external systems:
      - Google Gemini AI API (AI tutoring, roadmap generation, quiz creation)
      - YouTube Data API v3 (video resource discovery)
      - Reddit API (community content aggregation)
      - Google OAuth 2.0 (optional social login)
      - SMTP server (transactional email)

    The system does NOT integrate with:
      - Any payment gateway (v1.0 is free)
      - Any LMS (Learning Management System)
      - Any school or institution database

2.2 PRODUCT FUNCTIONS
    At the highest level, PrepCreatine performs these major functions:
      F1.  User registration, authentication, and onboarding
      F2.  AI-powered tutoring chat with streaming responses
      F3.  Personalized study roadmap generation and management
      F4.  Complete syllabus tracking with progress persistence
      F5.  Mock test creation, execution, and performance analysis
      F6.  Knowledge ingestion from URLs, PDFs, and text notes
      F7.  RAG-based source-grounded AI chat
      F8.  Student analytics dashboard
      F9.  Community Q&A with mentor participation
      F10. Study game mode for motivational support
      F11. Mentor oversight and student progress monitoring
      F12. Email notifications and in-app notifications

2.3 USER CLASSES

    2.3.1 STUDENT (primary user)
      Description: A person preparing for any examination. Age range 14–28
      primarily. Varying technical literacy. Uses the app daily for study
      planning, learning, and testing. May be highly motivated (JEE aspirant
      with 2 years of prep) or casually engaged (college student with an
      exam next week).
      Primary goals: Understand difficult concepts, know what to study next,
      track progress, test knowledge.
      Technical literacy: Medium — comfortable with mobile apps and web
      browsers. Not assumed to understand technical terms.

    2.3.2 MENTOR (secondary user)
      Description: A teacher, tutor, parent, or coach who oversees one or
      more students. Does not use the study features directly. Primarily
      monitors student progress and leaves guidance notes.
      Primary goals: See if a student is studying regularly, identify their
      weak areas, leave notes and encouragement.
      Technical literacy: Medium — may not be tech-savvy. Interface must
      require zero onboarding.

    2.3.3 ANONYMOUS VISITOR (landing page only)
      Description: A person who has not yet signed up. Arrives from search,
      social media, or a shared progress link.
      Primary goal: Understand what PrepCreatine does, decide to sign up.
      No access to app features.

2.4 OPERATING ENVIRONMENT
    Web: Any modern browser (Chrome 90+, Firefox 88+, Safari 14+, Edge 90+)
         on desktop (Windows, macOS, Linux) and mobile (iOS Safari, Android Chrome)
    Android: Android 8.0 (API level 26) and above
    Screen sizes: 320px wide (minimum) to 2560px wide (maximum)
    Network: Must function on 3G connections (1–3 Mbps) for core features.
             Streaming AI chat requires minimum 500kbps sustained.
             Syllabus tracker and offline features work at 0kbps (PWA cache).
    Server: Any Linux-based cloud host supporting Docker containers.
            Minimum 512MB RAM, 1 vCPU for development.
            Recommended production: 2GB RAM, 2 vCPU.

2.5 DESIGN AND IMPLEMENTATION CONSTRAINTS
    C1.  Backend must use Java 21, Spring Boot 4.0.x, Maven.
    C2.  Frontend must use React 18, Vite, TypeScript, Tailwind CSS.
    C3.  Database must be PostgreSQL 16 with pgvector extension.
    C4.  AI provider must be Google Gemini (primary). Model: gemini-1.5-flash.
    C5.  Authentication uses JWT stored in httpOnly cookies only.
         No localStorage for auth tokens under any circumstances.
    C6.  All AI prompts must produce deterministic output shapes (JSON schema
         enforced in prompt). No free-form AI responses for structured data.
    C7.  The syllabus data for all supported exams is embedded as static
         TypeScript constants in the frontend. It is never fetched from the
         backend. The backend only stores and retrieves user progress against
         topic IDs that originate from the frontend's static data.
    C8.  The system must function as a Progressive Web App (PWA) with offline
         support for the syllabus tracker, last saved roadmap, and cached notes.
    C9.  The Android app for v1.0 is a WebView wrapper. Full native Kotlin
         Compose implementation is a v2.0 deliverable.
    C10. The application must be deployable to Railway.app or Render.com
         without any infrastructure provisioning beyond a PostgreSQL instance
         and environment variable configuration.

2.6 ASSUMPTIONS AND DEPENDENCIES
    A1.  Google Gemini API free tier provides sufficient quota for the MVP
         (approximately 1,500 requests per day on free tier).
    A2.  YouTube Data API v3 free tier (10,000 units/day) is sufficient.
    A3.  Reddit API free tier (100 requests/minute) is sufficient.
    A4.  Students have access to a device with a modern browser.
    A5.  The JEE, NEET, CUET, GATE, CAT, UPSC, and State PET syllabi are
         stable enough to be embedded as static data (updated manually per
         official notification, not more than once per academic year).
    A6.  The hackathon demo requires a working web frontend with at minimum:
         landing page, sign up, onboarding, home, AI chat, roadmap, and
         one exam session. Full feature set is the production target.

3.1 AUTHENTICATION AND ACCOUNT MANAGEMENT
--------------------------------------------------------------------------------

FR-001  MUST  User Registration
  The system must allow any visitor to create an account using email and
  password. The system must validate that the email is not already registered,
  that the password meets minimum security requirements (8+ characters,
  at least one letter and one number), and that the user confirms their
  password. On successful registration, the system must set an httpOnly JWT
  cookie and redirect the user to the onboarding wizard.

FR-002  MUST  Email Verification
  After registration, the system must send an email verification link to the
  user's registered email address. The link must expire after 24 hours. The
  system must not block core functionality for unverified accounts but must
  display a dismissible banner prompting verification.

FR-003  MUST  User Login
  The system must allow registered users to log in with email and password.
  On success, the system must set a JWT httpOnly cookie and redirect to the
  home screen (or onboarding if not yet complete). The system must use an
  identical error message for both "email not found" and "wrong password"
  cases to prevent user enumeration.

FR-004  MUST  Google OAuth Login
  The system must offer "Continue with Google" as an alternative to email
  registration and login. On first Google login, the account is created
  automatically with onboardingComplete=false. On subsequent Google logins,
  the user is authenticated and redirected to home.

FR-005  MUST  Password Reset
  The system must provide a "Forgot password" flow: the user enters their
  email, receives a reset link (expires 1 hour), and can set a new password.
  The system must never reveal whether an email address is registered.

FR-006  MUST  Logout
  The system must allow users to log out from any authenticated page. Logout
  must clear the JWT cookie and redirect to the landing page.

FR-007  SHOULD  Account Deletion
  The system should allow users to permanently delete their account from the
  Settings page. Deletion must require the user to type "DELETE" as
  confirmation. All personal data must be removed within 24 hours of deletion.

FR-008  MUST  Session Persistence
  The JWT must remain valid for 7 days (MVP). The user must not be asked to
  log in again within 7 days of their last login. After expiry, the user is
  redirected to the login page with a message: "Your session has expired.
  Please log in again."

3.2 ONBOARDING
--------------------------------------------------------------------------------

FR-009  MUST  Exam Selection (Path A — Competitive Exams)
  On first login, the system must present a wizard. Step 1 must show a visual
  grid of exam cards: JEE, NEET, CUET, GATE, CAT, UPSC, State PETs, and
  "Something else." The user must select one. Selection must auto-advance to
  the next step after 400ms. "Coming soon" exam cards (dev/tech courses) must
  be non-clickable with a tooltip.

FR-010  MUST  Exam Date (Optional)
  Step 2 must offer three options: specific date (calendar picker), approximate
  month (grid), or "I'm not sure yet." All three options are fully valid. The
  step is skippable. No error must be shown if no date is provided. The date
  picker must prevent selection of past dates but must never block progression
  if somehow a past date is submitted.

FR-011  MUST  Weak Subject Selection (Optional)
  Step 3 must show subject chips relevant to the selected exam. The user may
  select any number (zero to all). This step is fully skippable. The system
  must store selected subjects as initial weak topics in the UserContext.

FR-012  MUST  Quick Setup (Optional)
  Step 4 must offer: daily study goal (slider, 30–300 minutes), mentor code
  (text input), and theme preference (light/dark/system). All three are
  optional. A failed mentor code lookup must never block onboarding completion.

FR-013  MUST  Quick Onboarding (Path C — School/Class/Semester)
  If the user selects "Something else" (school test, class test, semester),
  the system must show a 1-step onboarding asking only: what subject/topic
  are you studying (text input, required), and an optional URL for study
  material. The system must get the user to the AI chat in under 60 seconds.

FR-014  MUST  Back Navigation in Wizard
  The user must be able to go back to any previous onboarding step at any time.
  The back button must always be visible from step 2 onward. Going back to step
  1 shows an "Exit" option that returns to the landing page.

FR-015  MUST  Onboarding Idempotency
  If the onboarding API call fails, the wizard must retain all entered data and
  show a retry option. The system must never wipe the user's entered data on
  a network failure.

3.3 HOME SCREEN
--------------------------------------------------------------------------------

FR-016  MUST  Mode Cards
  The home screen must display five mode cards: "Want to learn", "Strategize",
  "Exam time", "Feeling low", and "Feed my notes". Each card must navigate to
  the corresponding feature on click. Cards must show a relevant icon in a
  circle, a title, and a brief description.

FR-017  MUST  Daily Stats Strip
  The home screen must display a summary strip showing: days until the exam
  (or a "Set exam date" link if not set), topics completed (count), and
  today's study progress toward the daily goal (minutes studied / goal).

FR-018  SHOULD  Streak Banner
  If the user has a study streak of 2 or more consecutive days, a banner must
  be shown on the home screen highlighting the streak count.

FR-019  MUST  Recent Activity
  The home screen must show the user's last 5 activities (test taken, topic
  marked done, AI question asked, source added) with description and timestamp.
  A skeleton must be shown while data loads.

3.4 AI TUTOR CHAT
--------------------------------------------------------------------------------

FR-020  MUST  Streaming Chat
  The AI chat interface must stream responses token by token using Server-Sent
  Events. The first token must appear within 500ms of sending a message. The
  user must be able to read the response as it generates without waiting for
  completion.

FR-021  MUST  UserContext-Aware Responses
  Every AI response must be informed by the user's current UserContext: exam
  type, study mode, weak topics, and exam date. The AI must adjust explanation
  depth based on study mode (in-depth = detailed, speed_run = bullet points
  and formulas only, overview = high-level, revision = quick recap).

FR-022  MUST  Concept Map Generation
  After every AI response, the system must generate and display an interactive
  concept map (directed graph) showing the relationships between concepts
  covered in the response. The concept map must use React Flow and must be
  interactive (pannable, zoomable, clickable nodes).

FR-023  MUST  YouTube Video Integration
  After every AI response, the system must display up to 5 relevant YouTube
  video cards below the AI message. Videos must show thumbnail, title, channel
  name, and duration. Thumbnails must load lazily.

FR-024  MUST  Conversation History
  The system must persist all chat conversations. The user must be able to
  view and resume past conversations from a sidebar list. Each conversation
  must display the first message as its title (truncated to 40 characters).

FR-025  MUST  RAG Mode — Source-Grounded Chat
  The user must be able to toggle "Use my notes" mode in the chat. When active,
  the AI must answer based on the user's uploaded study sources using vector
  similarity search. The AI must indicate which source excerpt supports its
  answer. The user must be able to restrict RAG to a specific source.

FR-026  MUST  New Conversation
  The user must be able to start a new conversation at any time. The new
  conversation must start with a clean context (no history from previous
  conversations).

FR-027  SHOULD  Chat Error Recovery
  If the AI stream is interrupted (network drop, API error), the system must
  show an inline error in the chat: "Connection interrupted. Please resend."
  with a resend button. The partially streamed response must remain visible.

FR-028  MUST  Message Sanitization
  All user messages must be sanitized before sending to the AI API. HTML tags,
  control characters, and null bytes must be stripped. Maximum message length
  is 4,000 characters. The UI must show a character counter when the user is
  within 500 characters of the limit.

3.5 STUDY ROADMAP
--------------------------------------------------------------------------------

FR-029  MUST  Roadmap Generation
  The system must generate a personalized study roadmap by combining: (1) the
  static syllabus graph structure from the frontend's exam syllabus data, and
  (2) AI-generated priority scores and adjusted time estimates per topic based
  on the user's UserContext. The AI is called once for priority data; the
  graph structure comes from static data. The combined roadmap is rendered as
  an interactive directed graph using React Flow.

FR-030  MUST  Topic Dependency Graph
  The roadmap must visually show prerequisite relationships between topics as
  directed edges. Foundational topics must appear above dependent topics in
  the graph layout (Dagre algorithm). The user must be able to pan, zoom,
  and click any node.

FR-031  MUST  Topic Status on Roadmap
  Each topic node must visually indicate its status: not started (gray),
  in progress (sky blue border), done (green border + strikethrough), or
  overdue (amber). Clicking a node must open the topic detail panel.

FR-032  MUST  Roadmap Filters
  The user must be able to filter the roadmap by: subject, status (all / not
  started / in progress / done), and priority level. Filters must apply
  instantly with no additional API call (pure frontend state filtering).

FR-033  MUST  Roadmap List View
  The user must be able to switch between graph view (React Flow) and list
  view (accordion tree grouped by subject). Both views must reflect the same
  data and the same filter state.

FR-034  SHOULD  Roadmap Regeneration
  The user must be able to request a fresh AI roadmap generation. Before
  regenerating, a confirmation dialog must appear. The regenerated roadmap
  must reflect any changes to the UserContext since the last generation.

FR-035  MUST  Topic Detail Panel
  Clicking any roadmap node must slide open a detail panel showing: topic
  name, subject badge, difficulty and estimated hours, status selector (3
  chips), subtopics checklist, YouTube videos for this topic, related source
  excerpts, and past test scores for this topic as a mini chart.

FR-036  MUST  Mark Topic Done from Roadmap
  Clicking "Done" in the topic detail panel must trigger a "Want to test your
  skills?" popup offering 3 difficulty levels. The user may also dismiss
  without testing. The topic status must update immediately (optimistic UI).

3.6 SYLLABUS TRACKER
--------------------------------------------------------------------------------

FR-037  MUST  Full Syllabus Tree
  The syllabus tracker must display the complete official syllabus for the
  user's selected exam as an expandable tree (subject → chapter → topic).
  The tree must load from the frontend's static syllabus data with zero API
  calls for structure. Only progress status is fetched from the server.

FR-038  MUST  Topic Completion Tracking
  The user must be able to check any topic as "not started", "in progress",
  or "done" by clicking a checkbox or status chip. Changes must be saved to
  the server via an optimistic update (immediate UI change, background sync).
  If the server save fails, the UI must roll back and show a toast.

FR-039  MUST  Progress Summary
  At the top of the syllabus tracker, a circular progress ring must be shown
  per subject displaying the completion percentage. The overall completion
  percentage must be computed as (done topics / total topics) × 100.

FR-040  MUST  Syllabus Search
  The user must be able to search topics by name in real time. The search must
  filter the tree as the user types with no server round-trip. The search must
  match substrings (e.g., "thermo" matches "Thermodynamics").

FR-041  MUST  Offline Syllabus
  The syllabus tracker must work offline. Progress changes made offline must
  be queued and synced when the network is restored. The offline queue must
  survive page refreshes (stored in IndexedDB by the service worker).

FR-042  MUST  Topic Done → Test Popup
  When a user marks a topic as done, a popup must appear: "Nice! Want to test
  your knowledge?" with three level buttons (Conceptual / Application /
  {ExamName} Level) and a "Not now" option. Selecting a level must navigate
  to the test configuration page with that topic pre-selected.

FR-043  MUST  Batch Progress Sync
  The system must support bulk syncing of up to 100 topic status changes in
  a single API request. This is used for initial import and offline sync replay.

3.7 MOCK TEST SYSTEM
--------------------------------------------------------------------------------

FR-044  MUST  Test Configuration
  Before starting a test, the user must choose from three test types: Full
  Mock (mirrors real exam structure), Topic-wise Practice (select subject +
  chapter + topic + difficulty), and Rapid Fire (10 questions, 30 seconds
  each). A difficulty level (1, 2, or 3) must be selectable.

FR-045  MUST  Exam-Style Interface
  The test interface must replicate the look and behavior of the actual JEE
  Main / NEET interface: one question at a time, countdown timer, section
  tabs (Physics / Chemistry / Mathematics or Biology), a question palette
  sidebar with color-coded status, mark-for-review functionality, and a
  save-and-next button. The sidebar and topbar of the main app must not be
  visible during a test session.

FR-046  MUST  Question Types
  The system must support three question types:
    a) MCQ (4 options, exactly one correct)
    b) Integer (numeric answer, no options shown)
    c) Multi-correct (4 options, one or more correct)
  Each type must have appropriate input controls.

FR-047  MUST  Timer Behavior
  For Full Mock tests: a countdown timer counts down from the exam duration
  (180 minutes for JEE, 200 minutes for NEET). For Rapid Fire: 30 seconds
  per question with individual question timers. When time expires, the test
  must be auto-submitted. The timer must turn red when less than 5 minutes
  remain for full mock tests.

FR-048  MUST  Navigation Warning
  If the user attempts to leave the test (browser back, refresh, close tab,
  navigate via sidebar), the system must show a browser confirmation dialog
  and a custom modal: "Your test is in progress. Leaving will abandon the
  test. Continue?" The test must not end unless the user confirms.

FR-049  MUST  Test Submission
  The "Submit Test" button must trigger a confirmation modal showing the count
  of answered and unanswered questions. On confirmation, the system must grade
  all answers, compute the score, and navigate to the results page.

FR-050  MUST  Test Results Page
  The results page must show: overall score and percentage, section-wise
  breakdown, a score trend chart comparing this test to previous tests for
  the same subject, and a detailed review of every question with the user's
  answer, the correct answer, and an expandable AI explanation (lazy-loaded
  per question on expand).

FR-051  MUST  Weak Topic Auto-Update
  After test submission, the system must automatically update the user's
  UserContext.weakTopics based on performance: topics where the user scored
  below 50% are added to weak topics; topics where the user scored above 80%
  are removed from weak topics. This update must trigger a roadmap cache
  invalidation.

FR-052  SHOULD  AI Question Generation
  When the question bank for a given topic and difficulty level has fewer than
  the required number of questions, the system must use the Gemini API to
  generate additional questions on demand. Generated questions must be stored
  in the question bank for future reuse.

FR-053  MUST  Test Abandonment
  If the user explicitly abandons a test (confirmation required), the test
  session must be marked as abandoned and no score must be recorded. The user
  must be returned to the test configuration page.

3.8 KNOWLEDGE INGESTION (MY NOTES)
--------------------------------------------------------------------------------

FR-054  MUST  URL Import
  The user must be able to import a web page by pasting its URL. The system
  must fetch the page content server-side, strip HTML to plain text, and
  process it. SSRF protection must be applied: private IP ranges must be
  rejected. The fetch must time out after 10 seconds. Maximum response size:
  500KB. Response must be returned immediately with status "pending"; processing
  completes asynchronously.

FR-055  MUST  PDF Upload
  The user must be able to upload a PDF file (maximum 20MB). The system must
  extract text using PDFBox. If the PDF is a scanned image and text extraction
  yields fewer than 100 characters, the system must return an appropriate
  error: "This PDF appears to be a scanned image. Text extraction is not yet
  supported." The file must not be permanently stored; only the extracted text
  is persisted.

FR-056  MUST  Text Paste
  The user must be able to paste raw text notes directly (minimum 50 chars,
  maximum 50,000 chars). Processing is synchronous (small enough to complete
  within the request timeout).

FR-057  MUST  Automatic Study Guide Generation
  For every successfully processed source, the system must asynchronously
  generate a study guide using the AI: a 3-sentence summary, a list of key
  concepts and topics, a formula sheet (if applicable), and 5 likely exam
  questions based on the content.

FR-058  MUST  Automatic Vector Embedding
  Every processed source must be split into overlapping 500-token chunks
  (50-token overlap) and each chunk must be embedded as a 768-dimensional
  vector using the Gemini text embedding API. These embeddings must be stored
  in PostgreSQL using the pgvector extension for subsequent similarity search.

FR-059  MUST  Source-Aware Chat (RAG)
  When the user enables "Use my notes" in the AI chat, the system must:
    a) Embed the user's query as a vector
    b) Find the 3 most semantically similar chunks from the user's sources
       using cosine similarity (pgvector)
    c) Inject these chunks into the AI system prompt
    d) Instruct the AI to answer from the chunks and cite them
  If no relevant chunks are found, the AI must say so and answer from general
  knowledge.

FR-060  MUST  Source Management
  The user must be able to view a list of all their imported sources with
  status (pending / processing / ready / failed), delete any source, and
  navigate to the source detail view.

FR-061  MUST  Source Viewer
  The source detail view must show three tabs: Study Guide (AI-generated
  content), Chat (source-specific AI chat), and Mind Map (React Flow graph
  generated from the source's key concepts). The mind map must be generated
  once on first view and cached.

3.9 ANALYTICS DASHBOARD
--------------------------------------------------------------------------------

FR-062  MUST  Summary Metrics
  The analytics dashboard must display four summary metric cards: topics
  completed (count), tests taken (count), average test score (%), and current
  streak (days).

FR-063  MUST  Activity Heatmap
  The dashboard must display a GitHub-style activity heatmap showing study
  activity for the last 365 days. Each cell represents one day; color intensity
  indicates minutes studied (gray = 0, sky blue gradient = 1 to maximum).
  Hovering a cell must show a tooltip with the date and minutes studied.

FR-064  MUST  Score Trend Chart
  The dashboard must display a line chart showing test score trends over the
  last 10 test sessions, with one line per subject. The chart must use
  Chart.js, be responsive, and include a custom legend.

FR-065  MUST  Subject Completion Chart
  The dashboard must display a horizontal bar chart showing completion
  percentage per subject (topics done / total topics).

FR-066  MUST  Weak and Strong Topics
  The dashboard must show two lists: "Areas to improve" (top 5 weakest topics
  by average test score) and "Your strengths" (top 5 strongest topics). Each
  item must show topic name, subject badge, average score, and a "Study now"
  link that opens the AI chat with the topic pre-loaded.

FR-067  MUST  Readiness Score
  The dashboard must display a circular gauge showing the user's readiness
  score (0–100) computed as:
    score = (completionPct × 0.50) + (avgTestScore × 0.35) +
            (daysRemainingFactor × 0.15)
  where daysRemainingFactor is a value between 20 and 80 based on days
  remaining. The score must be accompanied by a one-line interpretation.

FR-068  SHOULD  Mentor Notes Display
  If the user has a linked mentor, any notes left by the mentor must be
  displayed in the analytics dashboard in a "Mentor feedback" card.

FR-069  MUST  Public Progress Share Link
  The user must be able to generate a shareable read-only progress link from
  the Settings page. The link must display summary metrics and the activity

  a "Create your own PrepCreatine account" CTA.

FR-076  MUST  Game Generation

FR-078  MUST  Game Completion Screen

  or share the score. The share button must copy a text snippet to the clipboard.

FR-079  SHOULD  Weak Topic Update from Game
  After completing a game, if the user answered all questions for a concept
  correctly, that concept may be moved out of weak topics (at lower confidence
  than a full test). If answered incorrectly, the concept remains a weak topic.

SECTION 4: NON-FUNCTIONAL REQUIREMENTS

--------------------------------------------------------------------------------
4.1 PERFORMANCE
--------------------------------------------------------------------------------

NFR-001  Page Load Time
  The app shell (home screen skeleton) must be visible within 1.5 seconds on
  a 3G connection (1 Mbps) on first visit and within 300ms on repeat visits
  (served from service worker cache). Time to first contentful paint < 1.5s.

NFR-002  AI Chat Response Latency
  The first token of an AI streaming response must appear within 500ms of
  the user sending a message. This is the perceived latency — actual full
  response may take 3–6 seconds but users begin reading within 500ms.

NFR-003  API Response Time (non-AI)
  All non-AI API endpoints must respond within 300ms at the 95th percentile
  under normal load (up to 50 concurrent users). Database queries must
  complete within 100ms.

NFR-004  Roadmap Render
  The roadmap graph must render visually within 500ms of the data being
  received, even for exams with 100+ topics. React Flow virtualization and
  Web Worker graph layout must achieve this target.

NFR-005  Syllabus Tracker Load
  The syllabus tracker must render the full exam tree (300–500 topics) within
  200ms using TanStack Virtual for list virtualization.

NFR-006  Offline Functionality
  The syllabus tracker, last saved roadmap, and cached study guide must be
  accessible within 500ms when the device is offline (served from IndexedDB
  and service worker cache).

NFR-007  Bundle Size
  The initial JavaScript bundle (gzipped) must be under 100KB. Each lazy-
  loaded route chunk must be under 250KB gzipped. React Flow and Chart.js
  must be in separate chunks and loaded only when their pages are visited.

--------------------------------------------------------------------------------
4.2 RELIABILITY AND AVAILABILITY
--------------------------------------------------------------------------------

NFR-008  Uptime
  The system must target 99.5% monthly uptime (allowing approximately 3.6
  hours of downtime per month). Planned maintenance windows must be during
  off-peak hours (2–5 AM IST) and must not exceed 30 minutes.

NFR-009  Data Durability
  All user progress data must be written to PostgreSQL synchronously before
  acknowledging success to the frontend. No data loss is acceptable for topic
  completion and test score records.

NFR-010  Graceful Degradation
  If the Gemini API is unavailable, the system must:
    - Still allow syllabus tracking (no AI dependency)
    - Still display the roadmap from cached data
    - Still allow test-taking from the question bank (no AI generation)
    - Show a banner: "AI features are temporarily unavailable. We're working
      on it." in the chat, roadmap regeneration, and game pages.

NFR-011  Error Recovery
  Every React component that renders AI-generated or server-fetched content
  must be wrapped in an Error Boundary. An error in one section of the app
  must never crash the entire application.

NFR-012  Optimistic UI Consistency
  All optimistic UI updates (topic status, upvotes) must roll back gracefully
  on server failure within 5 seconds of the failed request.

--------------------------------------------------------------------------------
4.3 USABILITY
--------------------------------------------------------------------------------

NFR-013  Mobile Responsiveness
  Every page must be fully functional and visually correct on screens from
  320px (iPhone SE) to 2560px (4K desktop). No horizontal scrolling must be
  required at any viewport width. Tap targets must be at least 44×44px.

NFR-014  Onboarding Completion Time
  A first-time Path A user must be able to complete onboarding in under 3
  minutes. A Path C user must reach the AI chat in under 60 seconds.

NFR-015  Error Message Quality
  All user-facing error messages must be complete sentences in plain English,
  free of technical jargon, and actionable (tell the user what to do next).
  No ALL CAPS, no error codes, no stack traces.

NFR-016  Loading State Coverage
  No page, card, or data section may show blank white space while loading.
  Every loading state must be represented by a skeleton screen that matches
  the shape of the content that will appear.

NFR-017  Accessibility
  The application must meet WCAG 2.1 Level AA standards:
    - Color contrast ratio ≥ 4.5:1 for body text, ≥ 3:1 for large text
    - All interactive elements focusable via keyboard (Tab key)
    - All images have descriptive alt text (or empty alt for decorative)
    - Focus ring visible on all interactive elements
    - Screen reader: major content areas use semantic HTML
      (<main>, <nav>, <header>, <section>, <article>, <button>)

NFR-018  Consistent Navigation
  The user must never be trapped on a page with no exit. Every drill-down page
  (topic detail, test session, source viewer, game, thread view) must have a
  clearly labeled back navigation link.

NFR-019  Theme Persistence
  The selected color theme (light/dark/system) must apply immediately on
  selection and must be stored such that the correct theme is applied before
  the browser renders any visible content on subsequent visits (no flash of
  wrong theme).

--------------------------------------------------------------------------------
4.4 SECURITY
--------------------------------------------------------------------------------

NFR-020  Authentication Security
  JWT tokens must be stored only in httpOnly, Secure, SameSite=Strict cookies.
  Never in localStorage, sessionStorage, or any JavaScript-accessible storage.
  BCrypt password hashing with strength factor 12.

NFR-021  Input Sanitization
  All user-supplied text must be sanitized before storage and before display.
  HTML must be stripped using Jsoup (backend) and DOMPurify (frontend) wherever
  user content is rendered in the browser.

NFR-022  Rate Limiting
  The login endpoint: max 5 attempts per 15 minutes per IP.
  The signup endpoint: max 3 attempts per 10 minutes per IP.
  The AI chat endpoint: max 20 requests per minute per authenticated user.
  All other endpoints: max 120 requests per minute per IP.

NFR-023  SSRF Prevention
  The URL import feature must reject any URL that resolves to a private IP
  address range (10.x.x.x, 192.168.x.x, 172.16–31.x.x, 127.x.x.x, 169.254.x.x,
  ::1) or uses any protocol other than http or https.

NFR-024  API Key Protection
  All third-party API keys (Gemini, YouTube, Reddit) must be stored as server-
  side environment variables only. They must never appear in any frontend code,
  any API response, any log file, or any committed file.

NFR-025  HTTPS Enforcement
  All communications between the browser/app and the server must use HTTPS.
  HTTP connections must be redirected to HTTPS by the hosting platform.

NFR-026  CORS Policy
  The backend must only accept cross-origin requests from the configured
  frontend origin. Wildcard (*) CORS origins are forbidden.

NFR-027  Privacy of Test Data
  A mentor must only be able to view analytics for students who have explicitly
  linked to them via mentor code. No other students' data must be accessible
  to mentors. The backend must verify this link before returning any student
  data in mentor endpoints.

NFR-028  Data at Rest
  Passwords must never be stored in plain text. The raw text of uploaded sources
  (notes, PDFs) must be stored only in the authenticated user's account and
  must not be readable by other users or by mentor accounts.

--------------------------------------------------------------------------------
4.5 MAINTAINABILITY
--------------------------------------------------------------------------------

NFR-029  Code Structure
  Frontend: all business logic must be in hooks or utilities, never in
  component files. Components are presentation only.
  Backend: all business logic must be in service classes. Controllers are
  thin (validate → call service → return response). Repositories are pure
  data access.

NFR-030  Configuration Externalisation
  Every environment-specific value (API keys, database URL, CORS origin, cookie
  domain) must be in environment variables. No hardcoded values in source code.
  A .env.example file must be committed to the repository with placeholder
  values and descriptions of each variable.

NFR-031  Database Migrations
  All database schema changes must be made via Flyway migration files.
  Hibernate ddl-auto must be set to "validate" in all environments. Migration
  files must never be deleted or modified after being applied to any environment.

NFR-032  Logging Coverage
  All authentication events, all AI API calls, all external API calls, all
  background async tasks, and all errors must produce structured log entries
  with at minimum: timestamp, log level, request ID (MDC), user ID (if
  authenticated), and a human-readable message. Stack traces must never
  appear in the API response, only in server logs.

NFR-033  Test Coverage
  Service layer unit tests must achieve at least 60% line coverage as measured
  by JaCoCo. All Flyway migrations must be validated against a real PostgreSQL
  instance via Testcontainers in the CI pipeline.

--------------------------------------------------------------------------------
4.6 SCALABILITY
--------------------------------------------------------------------------------

NFR-034  Stateless Backend
  The backend must be stateless (no in-memory session state). All state is
  in the database or the JWT. Multiple backend instances must be deployable
  behind a load balancer without session affinity requirements.

NFR-035  Database Connection Pooling
  HikariCP must be configured with a maximum pool size of 20 connections.
  The application must not exhaust the database connection limit under
  50 concurrent users. Connection pool metrics must be available via
  Spring Actuator.

NFR-036  Caching Effectiveness
  YouTube API results must be cached for 24 hours (reducing YouTube API quota
  consumption by an estimated 90% for popular topics). Reddit API results
  must be cached for 30 minutes. Roadmap AI generation results must be cached
  for 1 hour per user per exam.

    - mentor_student_links   — mentor-student relationships
    - mentor_notes           — mentor notes per student
    - email_verification_tokens — auth tokens for email flows

8.2 DATA RETENTION
  DR-001: User account data is retained indefinitely while the account is active.
  DR-002: Upon account deletion request, all personal data must be deleted
          within 24 hours. This includes: user record, user context, topic
          progress, sources, source chunks, conversations, messages,
          test sessions, test answers, community posts, and notifications.
          Community answers may be anonymised (authorId set to null) rather
          than deleted to preserve thread integrity.
  DR-003: Email verification tokens expire after 24 hours. Password reset tokens
          expire after 1 hour. Expired tokens must be cleaned up by a scheduled
          job running daily.
  DR-004: Session logs (study activity) are retained for 2 years for analytics.
  DR-005: Test sessions and test answers are retained for 2 years.

8.3 DATA VALIDATION (GLOBAL RULES)
  DV-001: All text fields must have maximum length enforced both at the
          database (VARCHAR length) and at the API (Bean Validation) level.
  DV-002: Email addresses must be lowercase before storage.
  DV-003: All timestamps stored in UTC.
  DV-004: Exam dates must be in the future at the time of submission.
          If an existing exam date passes without update, it is treated as
          null (exam date unknown) — never as an error.
  DV-005: Topic IDs stored in user_topic_progress must match the format
          defined in the frontend's syllabus.ts (lowercase, hyphens only).
          The backend does not validate against a stored syllabus; it trusts
          the frontend-generated IDs.

SECTION 9: SECURITY REQUIREMENTS
================================================================================

The complete security implementation is specified in BSDD Section 5. This
section states the requirements independently of implementation.

SR-001  All API endpoints that return or modify user data must require a valid
        JWT token. The token must be read exclusively from the httpOnly cookie
        named "prepcreatine_token".

SR-002  Passwords must be hashed using BCrypt with a work factor of at least
        12 before storage. Plain text passwords must never be stored or logged.

SR-003  The system must not reveal whether a specific email address is registered
        when processing login failures or password reset requests. The response
        must be identical regardless of whether the email exists.

SR-004  All user-supplied text input must be sanitized to remove HTML tags,
        JavaScript code, control characters, and null bytes before storage
        and before rendering in any user interface.

SR-005  The URL import feature must validate that the target URL does not resolve
        to a private network address. The validation must be performed server-side.

SR-006  API keys for third-party services (Gemini, YouTube, Reddit) must only
        exist as server-side environment variables. They must never appear in
        client-side code, API responses, or version control.

SR-007  Rate limiting must be applied to all public-facing endpoints to prevent
        brute force and denial-of-service attacks.

SR-008  The CORS policy must only allow requests from the configured frontend
        origin. The wildcard (*) origin must never be used.

SR-009  A mentor must only be able to access data for students who have
        explicitly linked to their mentor code.

SR-010  The public progress share link (/progress/:token) must only expose
        aggregated statistics. It must not expose: individual question answers,
        source content, conversation history, or any other personally
        identifiable data beyond the student's name and exam type.

SECTION 10: ACCEPTANCE CRITERIA
================================================================================

The following criteria define when the system is considered ready for initial
deployment (hackathon MVP). All MUST requirements from Section 3 and all
NFR targets from Section 4 apply to full production readiness.

HACKATHON MVP ACCEPTANCE (minimum viable demo):
  AC-001: A new user can sign up, complete onboarding, and reach the home screen.
  AC-002: A user can ask the AI tutor a question and receive a streaming response
          with a concept map and YouTube video cards.
  AC-003: A user can view their personalized study roadmap as an interactive graph.
  AC-004: A user can take a mock test with at least 10 questions and see results.
  AC-005: A user can mark a syllabus topic as done and see progress update.
  AC-006: The landing page loads, displays the tagline "Creatine for your exam prep",
          and the signup CTA navigates to the signup page.
  AC-007: The application renders correctly on a mobile screen (375px width).
  AC-008: No unhandled JavaScript errors appear in the browser console during
          the above flows.

PRODUCTION READINESS ACCEPTANCE:
  AC-009: All FR-001 through FR-097 requirements are implemented and testable.
  AC-010: All NFR targets in Section 4 are measurably met under test load.
  AC-011: The Flyway migrations run cleanly on a fresh PostgreSQL 16 instance.
  AC-012: The backend starts successfully and passes the /actuator/health check.
  AC-013: The PWA service worker correctly caches the app shell and serves it
          offline (verified by Chrome DevTools → Application → Service Workers →
          Offline mode).
  AC-014: Sentry error monitoring receives at least one test event confirming
          the SDK is initialized correctly.
  AC-015: All environment variable names in .env.example are documented with
          descriptions and example values.
  AC-016: The "Brought to you by Rudra Agrawal" attribution appears in all
          locations specified in FDD Section 16.
